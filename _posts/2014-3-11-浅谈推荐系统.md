## Brief Content

所谓推荐系统，从字面上就能看出一二，它就是“用来做推荐的一组算法，或者一个成型的系统”。为什么需要推荐？难道我自己喜欢什么还不知道吗？有时候还真未必，尤其是在数据量巨大的情况下，用户想仅靠搜索引擎去精确定位某个 item 或 subject，往往会变得非常困难。更进一步说，很多用户甚至并不知道那些符合自己兴趣或需求的 item 的存在，这也是推荐系统（Recommender System，RS）存在的意义。

![](https://raw.githubusercontent.com/kakack/kakack.github.io/master/_images/140311-1.jpg)

常见的 RS 大致可以分成三类：

- 基于人：根据不同类型的人群进行推荐，粒度较粗。先将待推荐人的特征记录建模，再与已有模型进行比较，选择与其相似度较高的人群所感兴趣的 item 进行推荐。优点是对物品属性依赖较小，在用户历史行为不足时也有一定作用；缺点是方法比较粗放，个性化程度通常有限，实际效果往往一般。

![](https://raw.githubusercontent.com/kakack/kakack.github.io/master/_images/140311-2.jpg)

- 基于资源：这类方法把建模重点放在 item 上，思路与第一类相近，即先找出一个或几个相似度较高的 item，再将其推荐给对这些 item 感兴趣的人。它通常需要依赖 item 的历史记录或内容属性，因此在新物品场景下容易遇到冷启动问题，而且物品属性往往也难以完整罗列；但相对来说，它能更细致地刻画用户兴趣，推荐精度也可能更高。

![](https://raw.githubusercontent.com/kakack/kakack.github.io/master/_images/140311-3.jpg)

- 协同过滤（Collaborative Filtering, CF）：这是推荐系统中最经典、也最常见的一类方法。它还可以进一步分为基于用户的 CF、基于项目的 CF 和基于模型的 CF 三种；相比前两类方法，它通常能更充分地利用用户行为数据中的细粒度关系。

## Collaborative Filtering

概括来说，协同过滤是对一大群人和一大批待推荐的 item 进行分析：根据历史行为去寻找相似的用户或相似的 item，再据此形成一份 item-to-people 的推荐结果。整个过程大致可以分为 ***数据搜集、相似用户和 item 分类、进行推荐*** 三个步骤。

1. 数据搜集：一般有两种方式。一种是在线读取，例如用户对某项 item 做了某种感兴趣的操作，推荐系统便即时记录；另一种是将大量 log 文件存入数据库，定期读取。读取之后通常还要做过滤和清洗，例如去除操作数低于阈值的用户和 item，或者增加 black/white-list，对恶意操作和无意义操作进行清理。
2. 相似用户和 item 分类：最简单也最常用的相似度计算一般有两种，即欧几里得距离和皮尔逊相关度，另外还有 Cosine 相似度和 Tanimoto 系数等。这里的“分类”更准确地说，是按相似性进行比较和聚合，而不一定是严格意义上的监督分类。

   - 欧式距离：即常见的二维空间两点距离。相似度通常取其与 1 之和的倒数，因此取值范围在 0 到 1 之间，距离越近就认为二者越相似。一般来说，如果有多个共同标记内容，就把这些 item 的欧式距离求和后再取倒数。不过我在读到这种做法时有一个疑问：如果不考虑阈值，有 A、B、C 三个人，A 只对前三个 item 评分，分别为 {2,6,5}；B 对前五个 item 评分，分别为 {3,7,7,4,5}；C 也对前五个 item 评分，为 {2,6,5,4,4}。如果用简单的欧式距离计算，得到 A 和 B 的相似度是 1/(1+sqrt(pow(2-3)+pow(6-7)+pow(5-7)))=0.33333，而 B 和 C 的相似度是 1/(1+sqrt(pow(3-2)+pow(7-6)+pow(7-5)+pow(4-4)+pow(5-4)))=0.3090，结果反而是 A 与 B 更相似。但如果把图像画出来，直观上似乎还是 C 与 B 的曲线拟合程度更高。如果之后 B 和 C 又出现更多拟合度很高的评分，这种现象还会更加明显。所以我觉得，欧式距离对于评分 item 数量差异较大的用户之间的比较，效果未必理想，之后有机会还得继续研究。

   ![](https://raw.githubusercontent.com/kakack/kakack.github.io/master/_images/140311-4.jpg)![](https://raw.githubusercontent.com/kakack/kakack.github.io/master/_images/140311-5.gif)

   - 皮尔逊相关度：某种程度上，正是因为欧式距离有上述问题，才会让人想到从曲线拟合的角度来衡量相似性。皮尔逊相关度虽然比欧式距离复杂一些，但效果通常更好。一般做法是先找到两位评论者共同评价过的物品，再计算两者评分总和、平方和以及评分乘积之和，最后求得皮尔逊相关系数。其计算方式与方差的思路类似，返回值在 -1 到 1 之间，其中 1 表示对每样物品都有完全一致的评价。

   ![](https://raw.githubusercontent.com/kakack/kakack.github.io/master/_images/140311-6.gif)

   - 余弦相似度（Cosine-based Similarity）：将两人对某一组共 n 个 item 的评分看作一个 n 维向量，向量内积的余弦值即为相似度。

   ![](https://raw.githubusercontent.com/kakack/kakack.github.io/master/_images/140311-7.gif)

   - Tanimoto 系数：可以看作与内积计算相关的另一种相似度度量方式。

   ![](https://raw.githubusercontent.com/kakack/kakack.github.io/master/_images/140311-8.gif)

3. 进行推荐：在计算完相似度之后，就可以进一步进行推荐操作了。推荐可以以用户或 item 分别作为主要依据。前者是基于用户，根据不同 item 的评分找到相邻的 k 个用户，再预测当前用户尚未接触过的 item 并加以推荐。

![](https://raw.githubusercontent.com/kakack/kakack.github.io/master/_images/140311-9.gif)

后者则是找到相似度最近的 k 个物品，根据用户的历史偏好，预测当前用户尚未表现出偏好的 item 并进行推荐。

![](https://raw.githubusercontent.com/kakack/kakack.github.io/master/_images/140311-10.gif)

## Data Clustering

聚类是一种典型的无监督学习方法，没有标准答案。它的目的是将一个巨大而冗杂的数据集分割成若干个簇（cluster），使得同一簇内的元素相关程度较高，而不同簇之间的元素相关程度较低。在《集体智慧编程》一书中，介绍了用分级聚类来构建聚类树状图的方法，其思路是根据特征向量之间的距离来衡量相似性，将最近的两个族群合并为一个新的族群，以此类推，最终形成一个层次结构。

一般来说，提到聚类就不能不提 K-means 算法。它的核心思路在于找到某种“自然聚类”的中心，并将各个群组之间的均方误差降到最小。大致步骤如下：

![](https://raw.githubusercontent.com/kakack/kakack.github.io/master/_images/140311-11.jpg)

1. 如果我们要把整个数据集分成两个 cluster，那么先随机选择两个点作为中心位置；
2. 将数据集中所有的点依次分配给距离较近的那个“中心点”，从而得到两个簇；
3. 根据现有的分布情况，重新计算每个簇的中心位置；
4. 再根据新得到的中心点，把 1 到 3 步重新执行一遍，直到结果与上一次相同，或者误差低于某个阈值。
 
除了 K-means 外，Canopy 聚类算法也很常见。它的基本原则是：首先利用成本较低的近似距离计算方法，高效地将数据划分为多个组，这里称为一个 Canopy，我们姑且把它翻译为“华盖”，不同 Canopy 之间允许有重叠；然后再采用更严格的距离计算方式，精确计算同一 Canopy 中的点，并将它们分配到最合适的簇中。Canopy 聚类算法经常被用作 K-means 的预处理步骤，用来寻找较合适的 k 值和簇中心。

## Classification

分类在这里的作用，是将新获得的 item 归类到已有类别中的某一类里，常用的方法之一是决策树（Decision Tree）。

决策树有点像一张游戏地图：走到某个 NPC（树节点）面前，你的不同选择会决定接下来会走向哪个 NPC。不同于贝叶斯算法，决策树的构造过程并不依赖太多领域知识，而是通过属性选择度量，选出最适合将元组划分为不同类别的属性。所谓决策树的构造，本质上就是通过属性选择度量来确定各个特征属性之间的拓扑结构。

构造决策树的关键步骤是“分裂属性”。所谓分裂属性，就是在某个节点处按照某一特征属性的不同取值构造不同的分支，其目标是让各个分裂子集尽可能“纯”。这里的“纯”，是指尽量让一个分裂子集中的待分类项都属于同一类别。分裂属性大致可以分为三种情况：

1. 属性是离散值，且不要求生成二叉决策树。此时可用属性的每一个划分作为一个分支。

2. 属性是离散值，且要求生成二叉决策树。此时使用属性划分的一个子集进行测试，按照“属于此子集”和“不属于此子集”分成两个分支。

3. 属性是连续值。此时需确定一个值作为分裂点 split_point，并按照 `> split_point` 和 `<= split_point` 生成两个分支。
      
构造决策树的关键内容，就是进行属性选择度量。属性选择度量是一种选择分裂准则的方法，它试图将给定类标记的训练集合 D “尽可能好”地划分成若干类别，因此它会决定拓扑结构以及分裂点 split_point 的选择。属性选择度量算法有很多，一般采用自顶向下的递归分治法，并使用不回溯的贪心策略。常见的建树方法如 ID3，之后若有机会再详细整理，这里暂不展开。由于决策树具有较强的可解释性，其流程清晰可见；同时它又是非参数方法，能够处理连续变量和类别变量，因此在很多实际场景中都很受欢迎。
